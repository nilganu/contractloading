You are analyzing one or more pages of a hotel contract IMAGE.

Your ONLY job in this call is to identify the SKELETON of the rate matrix
visible on the page(s). Do not extract prices, do not produce hotel rows,
do not write notes. Just describe the dimensions.

LAYOUT PATTERNS — recognize which one applies before classifying columns.
A hotel contract page can use one of several rate-table shapes. Capture
which pattern you see by setting `layoutPattern` in the output.

  "board_stacked"     — Period column on the left, board column (BB/HB/…),
                        rooms across the top. Cells = per-person rate.
                        (Eg an Acrotel-style PDF.)

  "supplement_column" — Period column on the left, then a base-room price
                        column followed by supplement columns (SGL supp,
                        TPL reduction, child columns, room upgrade
                        columns, meal-upgrade columns, notes columns).
                        Single board per sheet (basic treatment).
                        (Eg a Volonline-style Excel sheet.)

  "period_column"     — Rooms DOWN the left (one room per row), periods
                        ACROSS the top (one column per period). The rate
                        type is "Per Room Per Night" (each cell is a
                        single per-room rate; same price applies for any
                        occupancy up to the room's Max Occupancy). A
                        "Max Occupancy" column usually sits next to the
                        room-name column.
                        (Eg a Salterra-style addendum PDF.)

For "period_column":
  - For each room row × period column, the same dollar amount fills DBL
    AND (if max occupancy permits) QDP / TPL — there is no SGL/DBL
    distinction because the rate is per-room not per-person.
  - The Max Occupancy column gives the room's Max Pax (and usually Max
    Adult equals it when no separate child policy is stated).
  - There is rarely a separate child policy; rate_type stays "Per Room
    Per Night" and child columns are typically absent.
  - Capture each room row as a `roomTypes` entry and each period column
    as a `periods` entry. Then the fill stage will emit
    rooms × periods Hotel rows.

- Detect the currency: EUR, USD, GBP, AED, EGP, or null if not visible.

  discount) for adding a third/fourth adult on an extra bed. Capture it

  cell text (e.g. "-30%", "free", "n/a", "20 EUR"). Many contracts call
  this "Extra Adult (extra bed) over 12yo" or similar; treat it
  separately from child policies. Returns null/empty if no such row

- For each child-policy row visible (e.g. ">1st child 0,1-11,99"), output

  AGE PRECISION: When a contract writes "0,1-11,99" or "0.1 - 11.99",
  capture the EXACT decimal — ageFrom = 0.1 (NOT 0), ageTo = 11.99
  (NOT 11 or 12). When the contract writes "2,0-11,99", ageFrom = 2

  convention). The dynamic column key downstream becomes

  matters.

  room cannot accept children — its perRoomLiteral entry for EVERY child
  position MUST be "n/a", regardless of what the rate table shows in the
  matching cell. The occupancy table is authoritative.

CRITICAL — hotel address vs supplier address:
The contract usually mentions TWO addresses:
- The HOTEL'S address (where guests stay). Often phrased as "based at <city>
  in <region>, <country>" or "the hotel is located at...". The HOTEL is the
  subject of the rates.
- The SUPPLIER / agent / tour-operator's address (the buyer's business
  location). Often phrased as "second party company ... based at City <X>,
  Country <Y>" or "tax office of <city>". This is where the agent lives,
  NOT where the hotel is.

The Moonstride row's "Country Code", "City / Area" and
"State / Province / Region" fields are ALL for the HOTEL. The "Supplier"
field is for the agent's company NAME only — no agent address fields are
exported. Put hotel address into hotelAddress.* and supplier name into
supplier. DO NOT cross-wire them.

For "supplier": look for a counter-party / buyer / agent company name in
the contract header. Common patterns:
  * "second party company with the name X" → supplier = X
  * "The undersigned X, hereinafter referred to as the Tour Operator..."
  * "Buyer: X" / "Travel Agent: X" / "On behalf of X"
  * A separate "company details" / "agent details" / "tour operator"

The supplier is NOT the hotel itself, NOT the chain, NOT the management
company. Return the raw company name as printed (e.g. "FLY 4 YOU SRL").
Return null only if no buyer/agent name is visible.

For "hotelAddress": look for where the hotel itself sits.
  * "based at <city> in <region>, <country>" → cityOrArea/stateOrRegion/countryCode
  * "The hotel is located at..."
  * Address block on the contract page near the hotel name.

for Italy, ES for Spain, etc). If only the country name is visible,

NEVER use the supplier's country/city for hotelAddress.
